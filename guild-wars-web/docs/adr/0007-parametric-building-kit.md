# 0007 — Parametric, themeable building kit

## Decision

City buildings gain a second production path alongside the fixed artist GLB models
(`h1`–`h19`): **Parametric Buildings** assembled at runtime from a **kit of small named
part slots**.

- **Procedural part kit, theme-swappable.** A Node tool (`tools/buildPartKit.mjs`,
  `npm run build:parts`) procedurally builds each part's geometry and exports a `.glb` per
  slot, plus one **texture atlas** (`atlas.png`) and a `manifest.json`, into
  `resources/buildingparts/<theme>/` (`default` = procedural placeholders). The generator
  references parts by **slot name** through the active theme, so the whole look can be swapped
  (per-district styles, or alternate settings) by pointing at a different theme folder. Parts
  are exported **geometry-only** (no embedded texture); the client applies one shared atlas
  material to all of them.
- **Bay-grid assembly.** A wall is a row of fixed-pitch **bays**. Rigid parts (posts, windows,
  doors, …) never scale; **flex infill panels X-scale** to fill a bay (the "60% bay"). Roof
  *shapes* are generated by the client (textured via an atlas region); only repeatable parts
  are GLBs.
- **Deterministic server-spec / seeded client-assembly.** The server emits a compact
  **Building Spec** (stored/broadcast, so identical for all players); the client assembler is
  a **pure, seeded function of the spec — no `Math.random`** — so every client builds
  byte-identical geometry.
- **Coexists** with the fixed models (which stay, used more sparingly). This pass ships the
  kit + atlas + generator + a standalone **gallery** preview (`buildingparts.html`); wiring
  parametric buildings onto plots is a later tranche.

## Why

Parametric assembly gives far more building variety than a fixed model set, lets building
materials tie into the cost/quality economy later, and — the main driver — makes the art
**themeable**: a theme is just a swapped set of part GLBs + atlas. Procedural placeholders
mean the kit's named slots exist now and can be refined or replaced by hand over time, exactly
as the fixed models were authored.

## Consequences

- New asset pipeline (headless `three` + `GLTFExporter` with a `FileReader` shim; a zlib PNG
  encoder for the atlas) and a new client generator + part library.
- The Building Spec schema and the kit's slot/grid schema (`manifest.json`) are now contracts
  shared by the tool, the generator, and any future theme.
- The "no `Math.random` in the assembler" rule is load-bearing for multiplayer consistency.

## Status

Accepted.

## Context for the three ADR criteria

- **Hard to reverse:** introduces an asset-generation pipeline, a kit/spec schema, a runtime
  generator, and a preview entry.
- **Surprising without context:** buildings are assembled from procedurally-exported part GLBs
  on a bay grid and textured from one atlas — a future reader expects fixed models.
- **Trade-offs:** kit vs fixed models; procedural-GLB kit vs runtime-only geometry; server-spec
  vs full-transform lists; bay-grid vs monolithic faces — each weighed and chosen.
