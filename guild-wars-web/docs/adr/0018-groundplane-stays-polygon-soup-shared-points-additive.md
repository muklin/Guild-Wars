The Groundplane keeps its current polygon-soup representation (each block, plot, street, gutter, and terrain plot is an independent polygon of `{x,y}` corners). A persistent shared **Point registry** is added *additively* — final surface vertices are deduplicated (by the same `toFixed(6)` snapping the generators already use transiently) into `points:[{id,x,y[,z]}]`, and each Surface gains a parallel `pointIds:[...]` reference — while the existing `blockCorners` arrays are retained as derived views so no downstream consumer changes.

The considered alternative was a full rewrite to an authoritative `Points` + `Surfaces` (shared-vertex / DCEL) model, with the generation pipeline rebuilt to author geometry topologically so adjacent surfaces share vertices by construction. It was rejected. The four standing contiguity bugs (dead-end plots not meeting the street; external/terrain plots reaching the road centreline; plots not meeting the gutter at corners; whole blocks missing) are **generation** bugs — wrong gutter-line placement, suppressed frontage seeds, and faces never traced — not **storage** bugs; a shared-vertex store fixes none of them on its own, because the code already reconstructs shared-vertex topology transiently via coordinate-key maps exactly where it matters. Only the full topological rebuild would structurally help, and only with the "missing blocks" case, at an estimated cost of weeks plus an unbounded geometric-degeneracy debugging tail (a hand-rolled planar-overlay/DCEL kernel, plus every `.blockCorners` consumer and the save/load path touched).

So the bugs are fixed in place in the existing generators, and the Point registry is introduced now — additively — purely as the foundation the future per-point z-height feature needs (and, as a bonus, as a contiguity diagnostic: points that are near-coincident but fail to merge reveal residual seams). The full topological authoring model remains deferred until the z-height work makes it worth its cost.
